# 🌿 Ветвление и Pull Request

## Зачем нужны ветки?

Ветки (branches) позволяют изолировать изменения от основной кодовой базы (`main`). Это необходимо для:
- Разработки новых функций без риска сломать рабочую версию.
- Исправления багов в отдельном окружении.
- Параллельной работы нескольких разработчиков (или агентов) над разными задачами.
- Безопасного тестирования изменений перед слиянием.

---

## 📋 Стратегии ветвления

Выбор стратегии зависит от размера команды, частоты релизов и требований проекта.

### 1. Git Flow (Классическая модель)
**Для кого:** Проекты с регулярными релизами, крупные команды.

**Структура веток:**
- `main` / `master` — стабильная версия, только релизы
- `develop` — основная ветка разработки
- `feature/*` — новые функции (ответвляются от `develop`)
- `release/*` — подготовка к релизу
- `hotfix/*` — срочные исправления (ответвляются от `main`)

**Преимущества:**
- Чёткая структура и разделение задач
- Подходит для сложных проектов с циклами релизов

**Недостатки:**
- Избыточна для небольших проектов
- Медленный цикл доставки

### 2. GitHub Flow (Упрощённая модель)
**Для кого:** Стартапы, проекты с непрерывным развертыванием (CI/CD).

**Принципы:**
- `main` — всегда готова к продакшену
- Любая новая задача → новая ветка (`feature/`, `fix/`)
- Ветка → PR → Code Review → Merge → Delete branch

**Преимущества:**
- Простота и скорость
- Минимум накладных расходов

**Недостатки:**
- Нет явной поддержки версионирования
- Требует зрелой культуры CI/CD

### 3. Trunk-Based Development (Разработка из одной ветки)
**Для кого:** Высокопроизводительные команды, DevOps-культура.

**Принципы:**
- Все разработчики работают в `main` (trunk)
- Короткоживущие ветки (< 1 день) или feature flags
- Частые коммиты (несколько раз в день)

**Преимущества:**
- Минимальные конфликты слияния
- Быстрая обратная связь

**Недостатки:**
- Требует высокой дисциплины
- Необходимы feature flags для незавершённых функций

## 🏷️ Правила именования веток

Стандартизированные префиксы помогают понять назначение ветки:

| Префикс | Назначение | Пример |
|---------|-----------|--------|
| `feature/` | Новая функциональность | `feature/add-user-authentication` |
| `fix/` или `bugfix/` | Исправление бага | `fix/login-error-on-mobile` |
| `hotfix/` | Срочное исправление в продакшене | `hotfix/critical-security-patch` |
| `docs/` | Изменения документации | `docs/update-api-docs` |
| `refactor/` | Рефакторинг без изменения логики | `refactor/simplify-payment-module` |
| `test/` | Добавление/изменение тестов | `test/add-unit-tests-for-cart` |
| `chore/` | Технические задачи (зависимости, конфиги) | `chore/upgrade-react-to-v18` |
| `style/` | Форматирование, стили (не влияют на логику) | `style/fix-indentation` |
| `perf/` | Улучшение производительности | `perf/optimize-image-loading` |

### Рекомендации по именованию:
- Используйте **нижний регистр** и **дефисы** (`kebab-case`)
- Будьте **конкретны**: `feature/add-login` лучше, чем `feature/new-stuff`
- Добавляйте **номер задачи**: `feature/PROJ-123-add-login`
- Избегайте специальных символов: `/`, `\`, `:`, `*`, `?`, `[`, `]`

## 🔧 Как создавать и управлять ветками

### 1. Создание новой ветки
Чтобы создать новую ветку и сразу переключиться на неё:
```bash
git checkout -b <имя-ветки>
# Или современный вариант (Git 2.23+):
git switch -c <имя-ветки>
```
*Пример:* `git switch -c feature/add-login`

### 2. Переключение между ветками
```bash
git checkout <имя-ветки>
# Или:
git switch <имя-ветки>
```

### 3. Просмотр списка веток
```bash
# Локальные ветки
git branch

# Все ветки (локальные + удалённые)
git branch -a

# Ветки с последним коммитом и автором
git branch -v

# Ветки, уже слитые в текущую
git branch --merged

# Ветки, НЕ слитые в текущую
git branch --no-merged
```
Текущая ветка будет отмечена звездочкой `*`.

### 4. Удаление веток
```bash
# Удалить локальную ветку
git branch -d <имя-ветки>        # Безопасное удаление (только если слита)
git branch -D <имя-ветки>        # Принудительное удаление

# Удалить удалённую ветку
git push origin --delete <имя-ветки>

# Удалить все локальные ветки, уже слитые в main
git branch --merged main | grep -v '\*\|main' | xargs -n 1 git branch -d
```

### 5. Переименование ветки
```bash
# Переименовать текущую ветку
git branch -m <новое-имя>

# Переименовать другую ветку
git branch -m <старое-имя> <новое-имя>

# Если ветка уже отправлена на сервер:
git push origin -u <новое-имя>
git push origin --delete <старое-имя>
```

## 🛠️ Продвинутые команды для работы с ветками

### 1. Stash — временное сохранение изменений
Полезно, когда нужно быстро переключиться на другую ветку, не коммитя текущие изменения.

```bash
# Сохранить изменения в stash
git stash

# Сохранить с сообщением
git stash save "WIP: работа над формой входа"

# Показать список stash
git stash list

# Применить последний stash (не удаляя)
git stash apply

# Применить и удалить
git stash pop

# Применить конкретный stash
git stash apply stash@{2}

# Удалить stash
git stash drop stash@{1}

# Очистить весь stash
git stash clear
```

### 2. Cherry-pick — перенос конкретного коммита
Переносит изменения из указанного коммита в текущую ветку.

```bash
# Перенести один коммит
git cherry-pick <commit-hash>

# Перенести несколько коммитов
git cherry-pick <hash1> <hash2> <hash3>

# Перенести диапазон коммитов (включительно)
git cherry-pick <hash-start>^..<hash-end>

# Отменить cherry-pick при конфликте
git cherry-pick --abort
```

**Когда использовать:**
- Хотфикс нужно применить в несколько веток
- Ошибка в выборе ветки для коммита
- Перенос конкретных фич между ветками

⚠️ **Предупреждение:** Cherry-pick создаёт новый коммит с другим хэшем, что может усложнить историю.

### 3. Работа с удалёнными ветками
```bash
# Получить список всех удалённых веток
git remote show origin

# Обновить информацию об удалённых ветках
git remote update origin --prune

# Создать локальную ветку из удалённой
git checkout -b <локальная-ветка> origin/<удалённая-ветка>
# Или автоматически (Git 2.x):
git checkout <имя-ветки>  # Git сам найдёт её в origin

# Отправить ветку на сервер и установить отслеживание
git push -u origin <имя-ветки>
```

## 🔄 Рабочий процесс (Workflow)

Наш стандартный процесс разработки выглядит так:

1. **Синхронизация с main**:
   ```bash
   git checkout main
   git pull origin main
   ```

2. **Создание ветки под задачу**:
   ```bash
   git switch -c fix/typo-in-readme
   ```

3. **Внесение изменений и коммит**:
   ```bash
   git add .
   git commit -m "fix: исправлена опечатка в README"
   ```

4. **Отправка ветки на сервер**:
   ```bash
   git push -u origin fix/typo-in-readme
   ```

5. **Создание Pull Request** → Code Review → Merge

6. **Очистка** (после мержа):
   ```bash
   # Удалить локальную ветку
   git branch -d fix/typo-in-readme
   
   # Удалить удалённую ветку (если не сделано через GitHub UI)
   git push origin --delete fix/typo-in-readme
   ```

## ⚠️ Частые ошибки и решения

| Проблема | Решение |
|----------|---------|
| **"error: src refspec ... does not match any"** | Ветка не существует локально или опечатка в имени |
| **"branch already exists"** | Выберите другое имя или удалите существующую ветку |
| **"not fully merged" при удалении** | Используйте `-D` для принудительного удаления, но убедитесь, что изменения сохранены |
| **Конфликты при переключении веток** | Закоммитьте или сделайте `git stash` перед переключением |
| **Ветка "отстала" от main** | Выполните `git merge main` или `git rebase main` в вашей ветке |

## 💡 Лучшие практики

✅ **Делайте:**
- Создавайте ветку от актуальной `main`
- Давайте понятные имена веткам
- Удаляйте ветки после слияния
- Часто синхронизируйтесь с `main`
- Делайте небольшие PR (до 400 строк)

❌ **Не делайте:**
- Не работайте напрямую в `main`
- Не создавайте долгоживущие ветки (> 1-2 недели)
- Не игнорируйте конфликты слияния
- Не пушьте чувствительные данные (пароли, ключи)

## 📊 Сравнение команд checkout vs switch

| Команда | Версия Git | Назначение | Примечание |
|---------|-----------|-----------|------------|
| `git checkout` | Все версии | Переключение веток + работа с файлами | Универсальная, но сложная |
| `git switch` | 2.23+ | Только переключение веток | Более понятная, рекомендуемая |
| `git checkout -b` | Все версии | Создать и переключиться | |
| `git switch -c` | 2.23+ | Создать и переключиться | Современный аналог |

> 💡 **Совет**: В нашем процессе ИИ-агент создает ветку, вносит правки и инициирует PR для утверждения изменений человеком.
