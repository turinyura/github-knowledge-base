# 🔀 Слияние веток и разрешение конфликтов

Слияние (merge) — это процесс объединения изменений из разных веток. Конфликты возникают, когда изменения в одной и той же части кода противоречат друг другу.

---

## 📊 Merge vs Rebase: Когда что использовать?

### Git Merge
**Что делает:** Создаёт новый "коммит слияния", который объединяет две ветки, сохраняя историю обеих.

**История коммитов:**
```
*   Merge commit (новый коммит)
|\  
| * feature-коммит 3
| * feature-коммит 2
* | main-коммит 2
|/  
* main-коммит 1
```

**Преимущества:**
- ✅ Сохраняет полную историю изменений
- ✅ Показывает, когда и какие ветки сливались
- ✅ Безопасен для публичных веток
- ✅ Легко откатить (просто `git revert`)

**Недостатки:**
- ❌ История может стать запутанной ("ёлочка")
- ❌ Много лишних коммитов слияния

**Когда использовать:**
- Слияние публичных веток
- Feature-ветки в main
- Когда важна полная история

### Git Rebase
**Что делает:** Переписывает историю, "переставляя" коммиты из одной ветки на конец другой.

**История коммитов:**
```
* feature-коммит 3 (перебазирован)
* feature-коммит 2 (перебазирован)
* main-коммит 2
* main-коммит 1
```

**Преимущества:**
- ✅ Чистая линейная история
- ✅ Нет лишних коммитов слияния
- ✅ Упрощает понимание истории

**Недостатки:**
- ❌ Переписывает историю (меняет хэши коммитов)
- ❌ Опасен для публичных веток
- ❌ Сложнее откатить

**Когда использовать:**
- Локальные feature-ветки перед мержем
- Обновление ветки от main
- Личные ветки, над которыми работаете только вы

### 📋 Сравнительная таблица

| Критерий | Merge | Rebase |
|----------|-------|--------|
| История | Сохраняется полностью | Переписывается |
| Коммиты | Добавляется merge-коммит | Нет лишних коммитов |
| Безопасность | Безопасен для всех веток | Только для локальных |
| Откат | Легко (`git revert`) | Сложно |
| Публичные ветки | ✅ Да | ❌ Никогда |
| Чистота истории | Средняя | Высокая |

> ⚠️ **Золотое правило rebase:** Никогда не делайте rebase для коммитов, которые уже отправлены в общий репозиторий и используются другими разработчиками!

## 🔷 Типы слияния

### 1. Fast-Forward Merge
**Что происходит:** Если целевая ветка не изменялась с момента создания вашей ветки, Git просто перемещает указатель вперёд.

**Схема:**
```
До:          После:
* main       * main (теперь здесь)
 \             \
  * feature     * feature
```

**Команда:**
```bash
git checkout main
git merge feature  # Автоматически fast-forward
```

**Принудительный fast-forward (запретить merge-коммит):**
```bash
git merge --ff-only feature
```

### 2. Three-Way Merge (Обычное слияние)
**Что происходит:** Создаётся новый коммит слияния, если ветки расходились.

**Схема:**
```
До:                    После:
* main                 * merge commit
 \                     / \
  * feature           *   * feature
                      main
```

**Команда:**
```bash
git merge --no-ff feature  # Принудительно создать merge-коммит
```

### 3. Squash Merge
**Что происходит:** Все коммиты из feature-ветки объединяются в один коммит при слиянии.

**Преимущества:**
- Чистая история без промежуточных коммитов
- Один коммит = одна фича

**Недостатки:**
- Теряется детальная история изменений

**Как сделать:**
```bash
# Через interactive rebase
git rebase -i HEAD~5  # Затем squash коммиты
git checkout main
git merge feature

# Или через GitHub UI при создании PR
```

## 🎯 Когда возникает конфликт?

Конфликт слияния происходит, если:
- Два разработчика изменили **одни и те же строки** в одном файле.
- Один разработчик изменил строку, а другой удалил её.
- Структура файла была изменена в обеих ветках несовместимым образом.

Git помечает такие места в коде специальными маркерами и требует ручного вмешательства.

### 🛡️ Превентивные меры для избежания конфликтов

1. **Часто обновляйте ветку от main:**
   ```bash
   git fetch origin
   git rebase origin/main  # или git merge origin/main
   ```

2. **Делайте небольшие коммиты** — легче разрешать конфликты

3. **Координируйте работу** — обсуждайте, кто какие файлы меняет

4. **Используйте .gitattributes** для бинарных файлов:
   ```
   *.png binary
   *.jpg binary
   ```

5. **Автоматизируйте тесты** — CI проверит до мержа

## 🛠️ Процесс слияния

### 1. Слияние через `merge`

Находясь в целевой ветке (например, `main`), выполните:

```bash
git checkout main
git pull origin main
git merge feature/my-branch
```

Если конфликтов нет, Git автоматически создаст коммит слияния.

**Опции merge:**
```bash
# Принудительно создать merge-коммит (даже при fast-forward)
git merge --no-ff feature/my-branch

# Быстрое слияние только если возможно (без merge-коммита)
git merge --ff-only feature/my-branch

# Слияние с сообщением
git merge -m "feat: добавлена новая функция" feature/my-branch

# Отмена слияния в процессе
git merge --abort
```

### 2. Слияние через `rebase` (альтернатива)

Перебазирование позволяет переписать историю, делая её линейной:

```bash
git checkout feature/my-branch
git fetch origin
git rebase origin/main
```

**Interactive rebase (редактирование истории):**
```bash
# Перебазировать последние 5 коммитов
git rebase -i HEAD~5
```

**В редакторе можно:**
- `pick` — оставить коммит как есть
- `reword` — изменить сообщение коммита
- `edit` — остановить для редактирования
- `squash` — объединить с предыдущим
- `fixup` — объединить, игнорируя сообщение
- `drop` — удалить коммит

После rebase вернитесь в main и сделайте merge:
```bash
git checkout main
git merge feature/my-branch
```

> ⚠️ **Важно**: Никогда не делайте `rebase` для общих (публичных) веток, над которыми работают другие люди.

## 🚨 Разрешение конфликтов

Если при слиянии возник конфликт:

### Шаг 1: Найдите файлы с конфликтами
Git выведет список файлов со статусом `both modified`:
```bash
git status
```

### Шаг 2: Откройте файл в редакторе
Вы увидите маркеры конфликта:
```text
<<<<<<< HEAD
Изменения из текущей ветки (main)
=======
Изменения из сливаемой ветки (feature)
>>>>>>> feature/my-branch
```

### Шаг 3: Отредактируйте код
Оставьте нужную версию или объедините логику вручную. **Удалите маркеры** `<<<<<<<`, `=======`, `>>>>>>>`.

### Шаг 4: Зафиксируйте разрешение конфликта
```bash
git add <имя_файла>
git commit -m "fix: разрешен конфликт слияния"
```
*(При обычном merge коммит часто создается автоматически после `git add`)*.

### Шаг 5: Проверьте результат
```bash
git log --oneline -5
git diff HEAD~1
```

## 🧰 Инструменты для разрешения конфликтов

### 1. Встроенные в редакторы кода

**VS Code:**
- Автоматически подсвечивает конфликты
- Кнопки "Accept Current", "Accept Incoming", "Accept Both"
- Расширение "Git Lens" для дополнительной информации

**IntelliJ IDEA / WebStorm:**
- Трёхпанельное сравнение (Your, Their, Result)
- Интеллектуальное слияние
- Предпросмотр результата

### 2. Специализированные инструменты

**Meld (Linux/Windows/Mac):**
```bash
# Установка
sudo apt install meld  # Linux
brew install meld      # Mac

# Настройка как инструмента по умолчанию
git config --global merge.tool meld
git config --global mergetool.meld.cmd "meld \"$LOCAL\" \"$BASE\" \"$REMOTE\""
git mergetool
```

**KDiff3:**
```bash
git config --global merge.tool kdiff3
git mergetool
```

**P4Merge (Perforce):**
```bash
git config --global merge.tool p4merge
git config --global mergetool.p4merge.cmd "p4merge \"$LOCAL\" \"$BASE\" \"$REMOTE\" \"$MERGED\""
```

### 3. Настройка mergetool по умолчанию
```bash
# Выбрать инструмент
git config --global merge.tool <instrument>

# Запустить инструмент
git mergetool

# Автоматически создавать backup
git config --global mergetool.keepBackup false
```

## 🤖 Пример из нашего процесса

При работе с ИИ-агентом конфликты могут возникнуть, если:
- Вы вручную изменили файл, который ранее редактировал агент.
- Агент обновил файл на основе устаревшей версии `main`.

**Алгоритм действий:**
1. Перед запуском агента всегда делайте `git pull`, чтобы актуализировать локальную ветку.
2. Если агент сообщил о конфликте при пуше, подтяните изменения: `git pull origin main`.
3. Разрешите конфликты локально, затем попросите агента продолжить или закоммитьте изменения вручную.

**Проактивная стратегия:**
```bash
# Перед началом работы с агентом
git checkout main
git pull origin main
git switch -c feature/agent-task

# После работы агента, перед пушем
git fetch origin
git rebase origin/main  # Решить конфликты сразу
git push -u origin feature/agent-task
```

## 🔍 Диагностика и отладка слияния

### Просмотр ожидаемых изменений
```bash
# Что изменится при слиянии
git diff main..feature

# Статистика изменений
git diff --stat main..feature

# Предварительный просмотр merge
git merge --no-commit --no-ff feature
git status  # Проверить конфликты
git merge --abort  # Отменить если что-то не так
```

### После проблемного слияния
```bash
# Посмотреть историю слияний
git log --merges

# Найти коммит, вызвавший проблему
git bisect start
git bisect bad  # Текущая версия плохая
git bisect good <хороший-коммит>
# Git будет предлагать коммиты для проверки
```

### Откат слияния
```bash
# Если слияние ещё не закоммичено
git merge --abort

# Если слияние уже закоммичено
git revert -m 1 <merge-commit-hash>
```
