# Терминология Git и GitHub

Полный справочник ключевых понятий с подробными объяснениями, назначением и методами работы.

---

## Repository (Репозиторий)

**Определение:** Хранилище проекта, содержащее все файлы, историю их изменений, ветки, теги и настройки.

### 🎯 Для чего предназначено
- **Централизованное хранение**: Все файлы проекта в одном месте с полной историей изменений
- **Отслеживание версий**: Возможность вернуться к любой предыдущей версии проекта
- **Совместная работа**: Несколько разработчиков могут работать над одним проектом без конфликтов
- **Резервное копирование**: Удалённый репозиторий на GitHub служит резервной копией вашего кода

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Создание локального репозитория | `git init` | Инициализация нового репозитория в текущей папке |
| Клонирование удалённого репозитория | `git clone <url>` | Копирование репозитория с GitHub на локальный компьютер |
| Проверка статуса | `git status` | Показывает изменённые файлы и текущую ветку |
| Просмотр истории | `git log` | Отображает историю коммитов |
| Добавление удалённого репозитория | `git remote add origin <url>` | Привязка локального репозитория к удалённому |

**Пример из реальной жизни:** Представьте библиотеку, где хранятся все книги (файлы проекта), а также журнал, в котором отмечены все изменения — кто взял книгу, когда вернул, какие были сделаны пометки на полях. Репозиторий — это как раз такая библиотека с полной историей всех действий.

## Commit (Фиксация/Коммит)

**Определение:** Логическая единица изменений, фиксирующая состояние файлов в определённый момент времени с уникальным хешем, автором, датой и сообщением.

### 🎯 Для чего предназначено
- **Сохранение прогресса**: Фиксация завершённых этапов работы
- **Документирование изменений**: Сообщение коммита объясняет, что и зачем было изменено
- **Возможность отката**: Можно вернуться к любому коммиту в истории
- **Code Review**: Позволяет ревьюерам понимать логику изменений

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Создание коммита | `git commit -m "сообщение"` | Фиксирует изменения с сообщением |
| Коммит с подробным описанием | `git commit` | Открывает редактор для многострочного сообщения |
| Изменение последнего коммита | `git commit --amend` | Позволяет исправить сообщение или добавить забытые файлы |
| Просмотр деталей коммита | `git show <hash>` | Показывает изменения в конкретном коммите |
| Отмена коммита (до push) | `git reset HEAD~1` | Удаляет последний коммит, оставляя изменения |

**Лучшие практики написания сообщений коммитов:**
- Используйте повелительное наклонение: "add feature", а не "added feature"
- Первая строка — до 50 символов, краткое описание
- При необходимости добавьте детальное описание после пустой строки
- Используйте префиксы: `feat:` (новая функция), `fix:` (исправление), `docs:` (документация), `refactor:` (рефакторинг)

**Пример из реальной жизни:** Это как сохранение черновика документа или фотографии с меткой времени. Например, вы пишете роман и делаете сохранение после каждой главы, чтобы всегда можно было вернуться к этой версии. Каждый коммит — это «сохранёнка» с описанием, что именно было изменено.

## Branch (Ветка)

**Определение:** Отдельная линия разработки, позволяющая работать с кодом независимо от других изменений, начиная с определённой точки истории.

### 🎯 Для чего предназначено
- **Изоляция функций**: Разработка новых возможностей без риска для стабильной версии
- **Параллельная работа**: Несколько разработчиков могут работать над разными задачами одновременно
- **Эксперименты**: Безопасное тестирование идей, которые можно легко отбросить
- **Исправление багов**: Создание хотфиксов для продакшена без прерывания основной разработки

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Создать новую ветку | `git branch <имя>` | Создаёт ветку без переключения |
| Создать и переключиться | `git checkout -b <имя>` или `git switch -c <имя>` | Создаёт ветку и сразу переключается на неё |
| Переключиться на ветку | `git checkout <имя>` или `git switch <имя>` | Переключение между ветками |
| Показать все ветки | `git branch` | Список локальных веток с отметкой текущей |
| Показать удалённые ветки | `git branch -r` | Список веток на удалённом репозитории |
| Удалить ветку | `git branch -d <имя>` | Удаляет локальную ветку (если слита) |
| Принудительное удаление | `git branch -D <имя>` | Удаляет ветку даже если не слита |

**Стратегии именования веток:**
- `feature/<название>` — для новых функций
- `fix/<название>` — для исправлений багов
- `hotfix/<название>` — для срочных исправлений в продакшене
- `docs/<название>` — для изменений документации
- `refactor/<название>` — для рефакторинга кода

**Пример из реальной жизни:** Представьте, что вы ремонтируете дом. Вместо того чтобы перестраивать комнату, пока живёте в ней, вы строите копию дома (ветку) и экспериментируете там. Если результат устраивает, переносите изменения в основной дом (слияние веток).

## Pull Request (Запрос на объединение/PR)

**Определение:** Механизм предложенных изменений в репозитории на GitHub, позволяющий обсудить код, проверить его и запросить слияние ветки с основной.

### 🎯 Для чего предназначено
- **Code Review**: Возможность для команды проверить качество кода перед слиянием
- **Обсуждение изменений**: Комментарии к конкретным строкам и общим решениям
- **Автоматические проверки**: Запуск тестов, линтеров и других CI/CD проверок
- **Документирование**: История того, какие изменения были внесены и почему
- **Контроль качества**: Гарантия, что в основную ветку попадает только проверенный код

### 🔧 Методы работы
| Действие | Как выполнить | Описание |
|----------|--------------|----------|
| Создание PR | Через интерфейс GitHub | Кнопка "Compare & pull request" после пуша ветки |
| Добавление ревьюеров | В интерфейсе PR | Назначение коллег для проверки кода |
| Внесение правок | Локально + push | После комментариев ревьюера исправьте код и сделайте push в ту же ветку |
| Одобрение изменений | Кнопка "Review changes" → "Approve" | Ревьюер подтверждает готовность к слиянию |
| Слияние PR | Кнопка "Merge pull request" | Объединяет изменения с целевой веткой |
| Закрытие без слияния | Кнопка "Close pull request" | Отклоняет изменения |

**Структура качественного PR:**
1. **Заголовок**: Кратко и понятно (например, "Добавлена авторизация через OAuth")
2. **Описание**: Что сделано, зачем, какие проблемы решает
3. **Скриншоты/видео**: Для визуальных изменений UI
4. **Чек-лист**: Что проверено перед отправкой
5. **Ссылки**: На связанные задачи или обсуждения

**Пример из реальной жизни:** Это как отправка рукописи редактору с просьбой проверить и принять изменения. Вы предлагаете свои правки, редактор может оставить комментарии, запросить доработки, и только после этого изменения будут приняты в основную версию книги.

## Merge (Слияние)

**Определение:** Процесс объединения изменений из одной ветки в другую, создающий новый коммит слияния.

### 🎯 Для чего предназначено
- **Интеграция изменений**: Перенос завершённой функциональности в основную ветку
- **Синхронизация веток**: Обновление_feature-ветки изменениями из main
- **Завершение работы**: Финальный этап работы над задачей в отдельной ветке

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Слияние ветки | `git merge <имя-ветки>` | Сливаем указанную ветку в текущую |
| Слияние без коммита | `git merge --no-commit <имя>` | Позволяет проверить результат перед фиксацией |
| Отмена слияния | `git merge --abort` | Отменяет слияние при возникновении проблем |
| Быстрая перемотка | Автоматически | Если нет расхождений, Git просто перемещает указатель |

**Типы слияния:**
- **Fast-forward**: Когда целевая ветка не изменялась, Git просто перемещает указатель
- **Three-way merge**: Когда есть расхождения, Git создаёт новый коммит слияния
- **Squash merge**: Все коммиты feature-ветки объединяются в один (через GitHub)

**Пример из реальной жизни:** Представьте две реки, которые сливаются в одну. Воды смешиваются, образуя единый поток. Так и изменения из разных веток объединяются в единую историю проекта.

## Rebase (Перебазирование)

**Определение:** Процесс переписывания истории коммитов путём переноса изменений из одной ветки на конец другой ветки.

### 🎯 Для чего предназначено
- **Линейная история**: Делает историю коммитов более понятной и прямой
- **Актуализация ветки**: Перенос вашей feature-ветки на последние изменения main
- **Очистка истории**: Объединение промежуточных коммитов перед слиянием

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Перебазирование | `git rebase <ветка>` | Переносит текущую ветку на указанную |
| Интерактивный rebase | `git rebase -i HEAD~n` | Позволяет редактировать последние n коммитов |
| Продолжение rebase | `git rebase --continue` | После разрешения конфликтов |
| Отмена rebase | `git rebase --abort` | Отменяет процесс перебазирования |

**⚠️ Важные предупреждения:**
- **Никогда не делайте rebase публичных веток**, над которыми работают другие люди
- Rebase переписывает историю, что может вызвать проблемы у других разработчиков
- Используйте rebase только для локальных feature-веток перед слиянием

**Пример из реальной жизни:** Представьте, что вы написали главу книги, но пока вы писали, автор изменил предыдущие главы. Rebase — это как переписать вашу главу так, будто вы начали писать её после всех изменений автора, а не параллельно.

## Conflict (Конфликт слияния)

**Определение:** Ситуация, когда Git не может автоматически объединить изменения из разных веток, потому что они затрагивают одни и те же строки кода несовместимым образом.

### 🎯 Почему возникают конфликты
- Два разработчика изменили **одни и те же строки** в одном файле
- Один изменил строку, другой удалил её
- Структура файла была изменена в обеих ветках несовместимо

### 🔧 Методы работы
| Шаг | Действие | Описание |
|-----|----------|----------|
| 1 | Обнаружение | Git сообщает о файлах со статусом `both modified` |
| 2 | Открытие файла | Найдите маркеры конфликта в коде |
| 3 | Разрешение | Отредактируйте код, выбрав нужную версию или объединив логику |
| 4 | Фиксация | `git add <файл>` + `git commit` |

**Маркеры конфликта в коде:**
```text
<<<<<<< HEAD
Изменения из текущей ветки (main)
=======
Изменения из сливаемой ветки (feature)
>>>>>>> feature/my-branch
```

**Процесс разрешения:**
1. Откройте файл с конфликтом в редакторе
2. Найдите маркеры `<<<<<<<`, `=======`, `>>>>>>>`
3. Решите, какую версию кода оставить (или объедините обе)
4. Удалите маркеры конфликта
5. Сохраните файл
6. Добавьте файл: `git add <имя_файла>`
7. Завершите слияние: `git commit`

**Профилактика конфликтов:**
- Чаще делайте `git pull` для актуализации локальной ветки
- Координируйте работу с коллегами над одними и теми же файлами
- Разбивайте большие задачи на мелкие независимые изменения

**Пример из реальной жизни:** Два редактора одновременно правят одну и ту же статью в Википедии. Один изменил абзац, другой написал совсем другой текст на том же месте. Нужно вручную решить, какая версия правильная, или объединить оба варианта.

## Remote (Удалённый репозиторий)

**Определение:** Версия репозитория, размещённая на сервере (GitHub, GitLab и др.), доступная через сеть.

### 🎯 Для чего предназначено
- **Синхронизация**: Обмен изменениями между разработчиками
- **Резервное копирование**: Хранение копии кода на внешнем сервере
- **Хостинг PR**: Площадка для создания Pull Request'ов и code review
- **CI/CD**: Запуск автоматических тестов и развёртывания

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Просмотр удалённых репозиториев | `git remote -v` | Показывает URL подключённых remote |
| Добавление remote | `git remote add <name> <url>` | Добавляет новый удалённый репозиторий |
| Получение изменений | `git fetch <remote>` | Загружает изменения, но не сливает |
| Загрузка и слияние | `git pull <remote> <branch>` | Fetch + merge за одну операцию |
| Отправка изменений | `git push <remote> <branch>` | Пушит локальные изменения на сервер |

**Обычные имена remote:**
- `origin` — основной удалённый репозиторий (откуда клонировали)
- `upstream` — оригинальный репозиторий (для форков)

**Пример из реальной жизни:** Удалённый репозиторий — это как облачное хранилище (Google Drive, Dropbox) для вашего кода, но с дополнительной возможностью отслеживать историю изменений и работать в команде.

## HEAD

**Определение:** Указатель на текущую ветку или конкретный коммит, с которым вы работаете в данный момент.

### 🎯 Для чего предназначено
- **Отслеживание позиции**: Показывает, где вы находитесь в истории проекта
- **Навигация**: Позволяет перемещаться между коммитами и ветками
- **Операции**: Многие команды Git работают относительно HEAD

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Посмотреть текущий HEAD | `cat .git/HEAD` | Показывает, на какую ветку указывает HEAD |
| Переместить HEAD | `git checkout <commit>` | Переключает HEAD на конкретный коммит |
| Ссылка на предыдущий коммит | `HEAD~1` | Родительский коммит текущего |
| Ссылка на несколько коммитов назад | `HEAD~n` | n коммитов назад от текущего |

**Состояния HEAD:**
- **Прикреплённый (attached)**: HEAD указывает на ветку (обычное состояние)
- **Откреплённый (detached)**: HEAD указывает напрямую на коммит (осторожно!)

**Пример из реальной жизни:** HEAD — это закладка в книге, которая показывает, на какой странице вы сейчас находитесь. Вы можете перемещать закладку вперёд и назад по страницам (коммитам).

## Staging Area (Индекс/Область подготовки)

**Определение:** Промежуточная область, куда вы добавляете изменения перед коммитом. Позволяет selectively выбирать, какие изменения включить в следующую фиксацию.

### 🎯 Для чего предназначено
- **Селективный коммит**: Возможность закоммитить только часть изменений
- **Подготовка**: Проверка того, что именно будет зафиксировано
- **Группировка**: Объединение логически связанных изменений

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Добавить файл в индекс | `git add <файл>` | Подготавливает файл к коммиту |
| Добавить все изменения | `git add .` | Добавляет все изменённые файлы |
| Просмотр индекса | `git status` | Показывает, что готово к коммиту |
| Убрать из индекса | `git reset <файл>` | Убирает файл из staging area |
| Просмотр различий | `git diff --staged` | Показывает изменения в индексе |

**Рабочий процесс:**
1. Вносите изменения в файлы (Working Directory)
2. Добавляете нужные изменения в индекс (`git add`)
3. Проверяете (`git status`, `git diff --staged`)
4. Делаете коммит (`git commit`)

**Пример из реальной жизни:** Представьте корзину покупок в онлайн-магазине. Вы добавляете товары (изменения) в корзину (staging area), просматриваете, что выбрали, и только потом оформляете заказ (коммит). Можно добавить или убрать товары перед финальным заказом.

## Tag (Тег)

**Определение:** Именованная метка, указывающая на конкретный коммит в истории. Обычно используется для обозначения версий релизов.

### 🎯 Для чего предназначено
- **Версионирование**: Обозначение релизов (v1.0.0, v2.1.3 и т.д.)
- **Быстрый доступ**: Легко вернуться к важной точке в истории
- **Документирование**: Явное обозначение значимых коммитов

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Создать лёгкий тег | `git tag <имя>` | Создаёт тег на текущем коммите |
| Созвать аннотированный тег | `git tag -a <имя> -m "сообщение"` | Тег с сообщением и датой |
| Показать теги | `git tag` | Список всех тегов |
| Переключиться на тег | `git checkout <тег>` | Переход к помеченному коммиту |
| Отправить теги на сервер | `git push origin --tags` | Публикация тегов на GitHub |

**Типы тегов:**
- **Lightweight (лёгкие)**: Просто указатель на коммит
- **Annotated (аннотированные)**: Содержат имя теггера, дату, сообщение (рекомендуется для релизов)

**Пример из реальной жизни:** Тег — это как закладка с названием в книге, которая отмечает важную главу или раздел. Например, "Версия 1.0" — это как закладка на странице, где закончена первая версия продукта.

## Fork (Форк)

**Определение:** Копия репозитория, созданная под вашим аккаунтом GitHub. Позволяет свободно экспериментировать с изменениями, не затрагивая оригинальный проект.

### 🎯 Для чего предназначено
- **Вклад в open-source**: Возможность предлагать изменения в чужие проекты
- **Эксперименты**: Безопасное тестирование идей на копии проекта
- **Создание ответвления**: Начало собственного проекта на основе существующего

### 🔧 Методы работы
| Действие | Как выполнить | Описание |
|----------|--------------|----------|
| Создать форк | Кнопка "Fork" на GitHub | Создаёт копию репозитория в вашем аккаунте |
| Клонировать форк | `git clone <ваш-форк-url>` | Копирует форк на локальный компьютер |
| Синхронизация с оригиналом | `git fetch upstream` + `git merge` | Обновляет форк изменениями из оригинала |
| Создать PR в оригинал | Через интерфейс GitHub | Предлагает изменения владельцу оригинала |

**Типичный workflow для вклада в open-source:**
1. Форкаете репозиторий на GitHub
2. Клонируете свой форк локально
3. Создаёте ветку для изменений
4. Вносите правки и пушите в свой форк
5. Создаёте Pull Request в оригинальный репозиторий

**Пример из реальной жизни:** Форк — это как взять рецепт блюда из общественной кулинарной книги, сделать свою копию страницы и экспериментировать с ингредиентами. Если получилось лучше, можете предложить автору оригинала включить ваш вариант в книгу.

## Cherry-pick

**Определение:** Применение конкретного коммита из одной ветки в другую без слияния всей ветки целиком.

### 🎯 Для чего предназначено
- **Выборочное применение**: Перенос только нужных изменений, а не всей ветки
- **Исправление багов**: Применение хотфикса в несколько веток одновременно
- **Отмена решений**: Копирование коммита, который был отменён в другой ветке

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Cherry-pick коммита | `git cherry-pick <hash>` | Применяет указанный коммит к текущей ветке |
| Cherry-pick диапазона | `git cherry-pick <start>^..<end>` | Применяет несколько коммитов подряд |
| Без коммита | `git cherry-pick -n <hash>` | Применяет изменения, но не создаёт коммит |

**⚠️ Предостережения:**
- Cherry-pick создаёт новый коммит с тем же содержанием, но другим хешем
- Может привести к дублированию кода в истории
- Используйте осторожно, предпочитая обычное слияние когда возможно

**Пример из реальной жизни:** Представьте, что у вас есть две тетради с заметками. Cherry-pick — это как переписать одну конкретную заметку из первой тетради во вторую, вместо того чтобы переписывать всю тетрадь целиком.

## Blame

**Определение:** Команда, показывающая, кто и в каком коммите изменил каждую строку файла.

### 🎯 Для чего предназначено
- **Поиск автора**: Узнать, кто написал конкретный участок кода
- **Понимание контекста**: Найти коммит, где было внесено изменение
- **Debugging**: Понять историю изменения проблемного кода

### 🔧 Методы работы
| Действие | Команда | Описание |
|----------|---------|----------|
| Blame файла | `git blame <файл>` | Показывает авторов каждой строки |
| Blame с деталями | `git blame -L <start>,<end> <файл>` | Только для указанных строк |
| Blame в интерфейсе | GitHub: кнопка "Blame" | Визуальный просмотр на сайте GitHub |

**Пример вывода:**
```
^a1b2c3d (Иван 2024-01-15 10:30:45 +0300 1) def calculate_total():
 a1b2c3d0 (Иван 2024-01-15 10:31:20 +0300 2)     return sum(items)
 b2c3d4e1 (Мария 2024-01-20 14:22:10 +0300 3) # Добавлен налог
```

**Пример из реальной жизни:** Blame — это как журнал посещаемости в библиотеке, где отмечено, кто и когда брал каждую конкретную книгу. Позволяет найти человека, который добавил или изменил определённую информацию.

---

> 💡 **Совет для ИИ-агентов**: При работе с кодом всегда понимайте контекст изменений. Используйте `git blame` для понимания истории кода, создавайте осмысленные коммиты и обязательно проходите через Pull Request для важных изменений.

## CI/CD (Continuous Integration / Continuous Delivery)

**Определение:** Практика автоматической интеграции и доставки кода, включающая автоматический запуск тестов, проверок и развёртывания при каждом изменении в репозитории.

### 🎯 Для чего предназначено
- **Автоматическая проверка**: Тесты запускаются до мержа, чтобы выявить ошибки на ранней стадии
- **Контроль качества**: Гарантия, что в основную ветку попадает только протестированный код
- **Ускорение разработки**: Автоматизация рутинных проверок экономит время команды
- **Быстрое развёртывание**: Автоматическая доставка кода на продакшен после успешных проверок

### 🔧 Методы работы
| Действие | Как выполняется | Описание |
|----------|----------------|----------|
| Настройка CI | Файл `.github/workflows/ci.yml` | Конфигурация автоматических проверок для GitHub Actions |
| Запуск тестов | Автоматически при push/PR | Система сама запускает тесты при изменениях |
| Проверка перед мержем | Branch protection rules | GitHub не позволит смержить PR, если тесты не прошли |
| Просмотр результатов | Вкладка Actions на GitHub | Логи и результаты всех проверок |

**Пример фразы:** «Автоматизируйте тесты — CI проверит до мержа» означает, что нужно настроить автоматическую систему проверки кода, которая будет запускаться перед тем, как новые изменения попадут в основную версию проекта. Вместо ручной проверки, специальные тесты (написанные заранее) будут автоматически проверять, всё ли работает правильно. Если что-то не так, система не позволит объединить код, пока ошибки не исправят.

**Преимущества CI/CD:**
- 🚫 **Раннее обнаружение ошибок**: Проблемы выявляются сразу, а не после мержа
- ⚡ **Быстрая обратная связь**: Разработчик узнает о проблемах через несколько минут
- 🔒 **Безопасность**: Нельзя случайно сломать основную ветку
- 📊 **Прозрачность**: Вся история проверок доступна команде

**Пример из реальной жизни:** Представьте конвейер на заводе: каждая деталь (изменение кода) проходит автоматическую проверку качества перед тем, как попасть в готовый продукт. Если деталь бракованная, конвейер останавливается и не пропускает её дальше. CI/CD — это такой же конвейер для вашего кода.
